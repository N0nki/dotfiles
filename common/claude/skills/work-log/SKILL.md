# 作業ログ

重要な実装・トラブルシューティング・意思決定を再現可能な記録として `.claude_work/work-logs/` に Markdown で残します。あとから状況を追跡し、別セッションが続きを再開できるよう、意図・根拠・状態スナップショットを揃えます。

## 発動条件

このスキルは 2 つのモードで発動する。

### 記録モード — 作業状態を新規記録 / 既存ログを更新

- 「記録して」
- 「作業ログが欲しい」
- 「ログに残して」
- 「作業内容を記録」
- 「ドキュメント化して」

### 再開モード — 別セッションで未完了ログから作業を再開

- 「前回の続きをやろう」
- 「未完了のログから再開して」
- 「中断したやつ再開」
- 「続きをやろう」

## ワークフロー

### 記録モード

#### 1. 保存ディレクトリの確認・作成

ディレクトリが存在するか確認：

```bash
ls -d .claude_work/work-logs/ 2>/dev/null
```

存在しない場合は作成：

```bash
mkdir -p .claude_work/work-logs
```

#### 2. 作業情報の収集

会話内容とリポジトリ状況から以下を取得：

- **基本情報**: タイトル / 目的 / 実施内容（時系列）/ 検証結果 / 残課題
- **frontmatter 用**（再開とインデックスのために必須）:
  - `status` の判定（completed / in-progress / blocked）
  - `branch`: `git branch --show-current`
  - `last_commit`: `git rev-parse --short HEAD`
  - `continues_from`: 同タスクの前回 work-log があれば
- **Resume 用**（中断時の再開のために必須）:
  - 次セッションが最初に着手すべき具体的なアクション 1 つ
  - 最初に Read すべきファイル（`path:line` 形式）
  - 再開前の準備コマンド（worktree 切替、サーバ起動など）

#### 3. ログファイルの生成

下記テンプレートに沿って Markdown を生成し、命名規則に従って保存。

### 再開モード

#### 1. 未完了ログの抽出

```bash
grep -l '^status: in-progress' .claude_work/work-logs/*.md
```

- `blocked` ステータスも対象に含めるかはユーザー意図に応じて判断
- 候補が複数なら、最新更新を既定としてユーザーに確認

#### 2. Resume セクションの読み取り

対象ログの先頭（frontmatter + Resume セクション）を Read。本文全体は必要に応じて。

#### 3. 準備コマンドの実行

Resume の「準備コマンド」を実行（worktree 切替、サーバ起動など）。

#### 4. 「次の一手」に着手

Resume の「次の一手」に従って作業を再開。

#### 5. 完了/中断時の更新（必須）

セッション終了時、同じログを以下のように更新する:

- 実施手順のチェックボックス（`- [ ]` → `- [x]`）
- frontmatter `status`（`in-progress` → `completed` または `blocked`、続行なら維持）
- frontmatter `last_commit` を最新コミットへ更新
- Resume セクションを次の中断地点向けに書き換え（`status: completed` なら省略可）
- 連鎖戦略については「注意事項 / 連続セッションでの追記方針」を参照

## ファイル命名規則

```
.claude_work/work-logs/{今日の日付}-{英語ケバブケースの作業タイトル}.md
```

**形式:**

- 日付: `YYYY-MM-DD` 形式
- タイトル: 英語のケバブケース（小文字、ハイフン区切り）
- 拡張子: `.md`

**例:**

- `2024-04-03-mongodb-data-import.md`
- `2024-04-03-fix-authentication-bug.md`

日付の取得：

```bash
date "+%F"
```

## 記載必須項目

### 0. Frontmatter (YAML)

ファイル先頭に必ず配置。別セッションが `grep -l 'status: in-progress'` で未完了ログを抽出するための機械可読メタデータ。

```yaml
---
status: in-progress # completed | in-progress | blocked
branch: feature/foo
last_commit: aeafd50
continues_from: 2024-04-02-prev-task.md # 任意。同タスクの前回ログがあるときのみ
---
```

**status の判定基準:**

| 値            | 条件                                             |
| ------------- | ------------------------------------------------ |
| `completed`   | 当初目的を達成し、未完了タスクなし               |
| `in-progress` | 主要作業に進捗があるが、次セッションで続きを行う |
| `blocked`     | 外部待ち（レビュー、外部 API、人）で進行不可     |

`continues_from` は同タスクの前回ログがある場合のみ記載。連続セッションを辿れるようにする。

### 1. タイトル

形式: `# YYYY-MM-DD 日本語タイトル`

例: `# 2024-04-03 MongoDB データインポート機能の実装`

### 2. Resume / 再開ポイント

**別セッションが最初に読む箇所。5 行以内で次の一手を集約する。** 本文を全部読まなくても着手可能にすることが目的。

含める内容:

- **次にやるアクション 1 つ**（最も具体的に。関数名・行番号レベルまで）
- **最初に Read すべきファイル**（`path:line` 形式で 1〜3 個）
- **再開前の準備コマンド**（worktree、ブランチ切替、サーバ起動など）

例:

```markdown
## Resume / 再開ポイント

- **次の一手**: `src/auth/middleware.ts:42` の `validateToken` 関数で JWT 署名検証を実装
- **準備コマンド**: `git checkout feature/jwt-auth && pnpm dev`
- **関連ファイル**:
  - `src/auth/middleware.ts:42` — 実装中の関数
  - `src/auth/types.ts:15` — `AuthToken` 型定義
  - `tests/auth/middleware.test.ts` — 失敗中のテスト 3 件
```

`status: completed` の場合は省略可（または「完了済み — 再開不要」と 1 行のみ）。

### 3. 概要

1-2 文で作業内容を要約。

### 4. 背景と目的

箇条書きで以下を記載：

- 期待する成果
- 解決したい課題
- 既知の制約条件

### 5. 実施手順

**完了/未完了がわかるチェックボックス付き番号リスト**で記載。

- 完了したステップ: `- [x]`
- 未完了/中断したステップ: `- [ ]`
- 中断地点には `← 中断中` などのマーカーを付け、Resume セクションの「次の一手」と整合させる
- 各ステップに使用したコマンド・編集ファイル・再現可能な粒度を含める

例:

````markdown
- [x] 1. 依存パッケージのインストール
  ```bash
  npm install mongodb mongoose
  ```
- [x] 2. データベース接続設定ファイル作成
  - `config/database.js` に接続文字列を設定
- [ ] 3. User モデルのバリデーション追加 ← **中断中**
  - `src/models/User.js` の `email` フィールドに正規表現バリデータを追加予定
- [ ] 4. 統合テスト追加
````

### 6. 結果と検証

以下を記載：

- 成果物
- 確認方法 / 使用したチェックコマンド
- テスト結果
- 未完了の場合: 現時点で確認できているところまでを正直に書く

### 7. 作成・変更ファイル

形式：

```markdown
- `path/to/file`: 変更概要
```

例：

```markdown
- `src/db/connection.js`: MongoDB 接続処理を追加
- `src/models/User.js`: User モデルを新規作成（バリデーションは未実装）
- `package.json`: mongodb, mongoose を追加
```

### 8. 判断・注意事項

- 検討した複数の案
- 採用しなかった理由
- 追加の注意点・リスク・制限事項

### 9. 次のアクション

未完了作業・フォローアップを**チェックボックス付き**で記載。Resume の「次の一手」より粒度が粗い、中長期のタスクを置く。

```markdown
- [ ] User モデルのバリデーション実装を完了
- [ ] 認証エンドポイントの統合テスト追加
- [ ] パフォーマンス計測（10万件投入時）
```

## 記載ルール

### セキュリティ

**絶対に記載禁止:**

- パスワード / 秘密鍵 / API キー / アクセストークン / 個人情報

**安全な記載方法:**

- 環境変数名で表現: `<ENV_VAR>`、`$DATABASE_PASSWORD`
- マスク表記: `****`、`<REDACTED>`
- プレースホルダー: `<YOUR_API_KEY>`

例：

```markdown
❌ 悪い例: export API_KEY="sk_live_abc123def456"
✅ 良い例: export API_KEY="<YOUR_API_KEY>"
```

### フォーマット

- 見出しレベルを統一
- コードブロックには必ず言語を指定
- 表やリストは読みやすさを優先

### 言語

- 本文は日本語
- コマンド・ファイル名・設定キーは原文のまま
- 一般的な技術用語は英語のまま（無理にカタカナ化しない）

## 実行例

### 例1: 中断時のログ作成（次セッション再開を意識）

**ユーザー:** 「ここまで進んだので一旦記録して。続きは明日やる」

**Claude の対応:**

1. 現在の状態を取得：
   ```bash
   git branch --show-current
   git rev-parse --short HEAD
   git status --short
   ```
2. frontmatter に `status: in-progress`、`branch`、`last_commit` を埋める
3. **Resume セクション**を最優先で書く（次に着手する関数/ファイル、準備コマンド）
4. **実施手順**は完了分 `[x]`、中断分 `[ ]` で明記し、中断地点に `← 中断中` マーカー
5. ファイル名: `2024-04-03-mongodb-data-import.md`

### 例2: 完了タスクの記録

**ユーザー:** 「認証バグを修正したので記録して」

**Claude の対応:**

- frontmatter は `status: completed`
- Resume セクションは省略可（または「完了済み — 再開不要」と 1 行のみ）
- 実施手順はすべて `[x]`

## 注意事項

### ファイル上書き

同じ日付・タイトルのファイルがある場合：

1. ユーザーに確認
2. 上書きするか、別名（末尾に `-2` など）で保存

### 連続セッションでの追記方針

同じタスクを再開して進めた場合の選択肢：

1. **既存ログ更新**（推奨）: チェックボックス更新 / Resume 書き換え / frontmatter `status` 更新。タスクの最終状態が常に 1 ファイルに集約される。
2. **新規ログ + 連鎖**: 新ログを作り frontmatter `continues_from: <前回ログ>` で繋ぐ。日次の進捗を残したい場合に有効。

迷う場合はユーザーに確認。

### 機密情報の確認

ログに含まれる内容にパスワードや API キーなどが含まれていないか、保存前に必ず確認。
